# 道の駅リサーチスキル

次に訪問する道の駅の下調べを行い、`stations/YYYY-MM-DD_道の駅名/research.md` を作成・記入する。

## 手順

### 1. 引数の確認

`$ARGUMENTS` から **道の駅名** と **訪問日** を読み取る。

- **どちらか一方でも指定がない場合は、推測で進めず AskUserQuestion でユーザーに聞き直すこと。**
  - 道の駅名が不明 → 「どの道の駅をリサーチしますか？」と質問する
  - 訪問日が不明 → 「訪問日はいつですか？」と質問する（候補として直近の週末の日付を提示してよい）
- 訪問日は `YYYY-MM-DD` 形式に正規化する（例:「7月20日」→ 現在日付以降の直近の該当日）
- 道の駅名は正式名称を確認する（例:「桶川」→「道の駅 べに花の郷おけがわ」）。フォルダ名には地名などの短い通称を使ってよい

### 2. 選定基準のチェック

リサーチ開始時に、READMEの選定基準を満たすか確認する。

- 最寄り駅から徒歩90分以内が目安（バス併用ルートがあれば歓迎）
- 大きく超える場合は、その旨をユーザーに伝えて続行するか確認する

### 3. フォルダ作成

```bash
mkdir -p stations/YYYY-MM-DD_道の駅名
cp _template/*.md stations/YYYY-MM-DD_道の駅名/
```

- フォルダが既に存在する場合は上書きコピーせず、既存ファイルを更新する

### 4. Webリサーチと research.md の記入

WebSearch / WebFetch を使って調査し、テンプレートの全項目を埋める。

**情報源の優先順位（一次情報を優先）:**

1. 道の駅の公式サイト・公式SNS
2. 全国「道の駅」連絡会（michi-no-eki.jp）・関東「道の駅」公式（kanto-michinoeki.jp）の登録ページ
3. 所在市町村の公式サイト（開業日・アクセス・防災機能・地域の歴史）
4. 運営会社のプレスリリース
5. 観光情報サイト・ニュース記事・ブログ（補助情報として。価格や混雑状況など）

**調査する項目:**

- 基本情報：所在地、開業日（県内何番目か）、コンセプト、営業時間（店舗別も）、定休日、公式サイト、立地（沿線道路・防災機能）
- アクセス：最寄り駅、駅からの距離・徒歩所要時間、バス代替ルート（系統名・下車バス停・徒歩分数）、最寄りIC、駐車場（台数・料金）
- 特産品・グルメ：名物メニュー、店舗一覧、価格（判明する範囲で。変動する旨も記す）
- 見どころ・地域の取り組み：地域の歴史・特産の背景。**動画のナレーションで語る歴史ネタは裏取り元（公式情報源）を【〜で確認済み】の形で明記する**（候補を広く出して人間に選んでもらう詳細な手順はステップ5を参照）

**記入時のルール:**

- 調べても分からなかった項目は空欄にせず「要確認」と書き、末尾の注記に現地確認事項としてまとめる
- 情報源によって数値が食い違う場合は両方を併記し、「現地で確認」と添える
- 訪問日に影響する情報（臨時休業・工事・イベント・混雑傾向）を見つけたら ⚠️ 付きで目立つように書く
- 「参考リンク」セクションに使用した情報源をすべてリンク付きで残す

記入例として `stations/2026-07-04_桶川/research.md` を参考にすること。

### 5. 地域の歴史・トリビア候補出しと選定

台本のナレーションで語れる「意外な歴史・地域の豆知識」を、人間が選べる形で複数候補提示する。単に research.md の空欄を埋めるのではなく、採用されなかった候補も含めて広く調べ出すことが目的。

1. `docs/learnings.md` が存在すれば読み込み、「リサーチ：トリビア候補の選定傾向」の「採用されやすい観点」「不採用になりやすい観点」を確認してから候補を考える（初回など該当ログがなければスキップしてよい）
2. ステップ4と同じ情報源の優先順位（公式サイト→道の駅連絡会→自治体→運営会社→観光/ニュース記事）で、地名の由来・産業の変遷・地元の言い伝え・数字で語れる意外な事実などを幅広く調べる
3. 裏取りできた候補を5〜10件出す。各候補には以下を添える
   - 出典（【〜で確認済み】の形で明記。裏取りできないものは候補に含めない）
   - タグ：面白さ・意外性の見立て／ルート上のどこで話せそうか（駅出発直後・徒歩中盤・道の駅到着後など）
4. 候補を全てチャットに番号付きで提示し、ユーザーに採用・不採用を選んでもらう（候補数が多くAskUserQuestionの選択肢上限（4件）に収まらないため、AskUserQuestionは使わず自由回答で選定してもらう）
5. ユーザーの選定結果を `research.md` の「見どころ・地域の取り組み」セクションに反映する（採用分のみ本文に残す。不採用分は書かない）
6. `docs/learnings.md` の「リサーチ：トリビア候補の選定傾向」個別ログに、今回の道の駅名を見出しにして採用・不採用の内容とユーザーが添えた理由（あれば）を追記する
7. 個別ログ全体を見返し、同じ傾向の判断が2回以上の回で確認できた場合のみ「採用されやすい観点」「不採用になりやすい観点」を更新する

### 6. 徒歩ルート地図画像の取得

claude-in-chrome を使い、最寄り駅から道の駅までの徒歩ルート地図を OpenStreetMap 上でスクリーンショットし、`assets/route-map-osm.png` として保存する。

1. `mkdir -p stations/YYYY-MM-DD_道の駅名/assets` でフォルダを用意する
2. claude-in-chrome の関連ツール（tabs_create_mcp, navigate, computer, find など）をロードする
3. 新しいタブで `https://www.openstreetmap.org/directions` を開く
4. 出発地欄に最寄り駅名（例：「JR桶川駅」）、到着地欄に道の駅の正式名称または住所を入力する
5. ルート種別を「徒歩（Foot）」に切り替える
6. ルートが描画されるまで待ち、ブラウザウィンドウを 1280x720 程度にリサイズする（`resize_window`）
7. 地図全体が収まるように `computer` のスクリーンショット機能でキャプチャし、`stations/YYYY-MM-DD_道の駅名/assets/route-map-osm.png` として保存する
8. ルートが意図通りに描画されない（経路が見つからない・大きく迂回するなど）場合は、その旨をユーザーに報告し、手動確認を促す

**注意:**
- ダイアログ（alert/confirm等）を発生させる操作は避ける
- 取得できた画像がある場合のみ完了報告に含める。取得に失敗した場合は「地図画像は要確認」として報告する

### 7. サイト詳細画面の地図アニメーション用ウェイポイントの生成

道の駅詳細ページ（`website/app/components/RouteMap.tsx`）は、最寄り駅と道の駅を結ぶピンを道なりにアニメーションさせる。この経路（道路形状の座標列）を `data/stations.json` の `accessByTrain.routeWaypoints` に追加する。

1. `data/stations.json` から対象道の駅の `accessByTrain.nearestStationCoordinates`（出発地）と `coordinates`（到着地）を取得する
2. OSRM の公開ルーティングAPI（徒歩プロファイル）に問い合わせて実際の道路形状を取得する
   ```
   curl "https://router.project-osrm.org/route/v1/foot/{出発地lng},{出発地lat};{到着地lng},{到着地lat}?overview=full&geometries=geojson"
   ```
   - 返ってきた `distance`（m）が route.md 記載の距離と大きく食い違う場合でも、既存の座標（マーカー位置）に対する道路追従ルートとして地図表示上は問題ない。route.md側の数値を修正する必要はない
3. 取得した座標列（数十〜100点超）を Ramer-Douglas-Peucker法などで間引き、15〜25点程度に単純化する（点が多すぎるとJSONが肥大化するため）
4. 間引いた座標列を `{ "lat": ..., "lng": ... }` の配列として `accessByTrain.routeWaypoints` に追加する（出発地・到着地自体は `nearestStationCoordinates`/`coordinates` で既に表現されているため含めない）
5. `npm run typecheck`（`website/`ディレクトリ）でエラーがないことを確認し、可能であれば dev サーバーで詳細ページの地図を目視確認する

**注意:**
- OSRM公開デモサーバーはレート制限があるため、1駅あたり1回のリクエストに留める
- ウェイポイントが無くても `RouteMap` は起点・終点の直線アニメーションにフォールバックするため、取得に失敗した場合は無理に追加せず「要確認」として報告してよい

### 8. 完了報告

- 調査結果の要点（営業時間・アクセス・名物・撮影上の注意）をまとめて報告する
- 「要確認」として残した項目を列挙する
- 地図画像（`assets/route-map-osm.png`）の取得可否を報告する
- `routeWaypoints` の追加可否を報告する
- 採用したトリビアと、`docs/learnings.md` に追記した内容（個別ログ・傾向の更新有無）を報告する
- 次のステップ（route.md の作成 = 徒歩ルートの詳細化、script.md の作成は `script-station` スキル）を案内する

## スコープ外

- route.md の詳細な徒歩ルート作成（区間ごとの表など）・script.md の台本作成はこのスキルでは行わない（route.mdはユーザーから別途依頼されたときに行う。script.mdは `script-station` スキルの担当）
- コミット・プッシュ・PR作成は、ユーザーの指示があるまで行わない
