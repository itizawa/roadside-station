# 進捗管理（予実）スキル

週1回・2026-07-31週から開始する動画投稿の「予定」と「実績」を突き合わせ、`docs/progress.md` を更新する。

## 管理方針（前提）

このプロジェクトでは以下の役割分担で予実を管理する。

- **各道の駅の進捗（リサーチ・ルート・台本・編集ステータス・実績公開日）のSSOTは[`data/stations.json`](../../../data/stations.json)の各stationオブジェクトの`pipeline`フィールド**（`research`/`route`/`script`/`editing`/`publishDate`）に置く。理由：`website/`側の表示データもここから計算されており、`research.md`等をdiffで比較する旧方式より二重管理にならない
- **「予定」（訪問日・編集作業日・投稿目標日時）はGoogleカレンダー**で管理する（`schedule-station` スキルが登録）。理由：リマインダー通知が必要な情報はカレンダーに向いている
- `docs/progress.md` は上記2つ（`data/stations.json`の`pipeline` とGoogleカレンダーの予定）を突き合わせた **派生ダッシュボード**。本スキルが再生成するものであり、手動加筆やGit管理のファイルそのものをSSOTとはしない
- 本スキルは両者を突き合わせて **1枚のダッシュボード（`docs/progress.md`）** に集約し、遅延がないかを可視化する。カレンダー側の作成・変更は行わない（`schedule-station` の担当）

## 手順

### 1. 対象の確認

`data/stations.json` の `stations` 配列を読み込む。`$ARGUMENTS` に道の駅名があれば `title` または `slug` が一致する1件のみ、なければ全件を対象にする。

### 2. 各stationの進捗判定（`data/stations.json`を参照）

対象の各station オブジェクトについて、以下のフィールドをそのまま使う（`stations/`配下のファイルをdiffで比較する必要はない）。

- `visitDate` / `visitStatus` → 訪問日・訪問ステータス
- `pipeline.research` / `pipeline.route` / `pipeline.script` → それぞれ `done`（済）/ `todo`（未）/ それ以外の値（例：`draft_feedback_pending`＝初稿・フィードバック待ち、`in_progress`＝進行中）はその値のニュアンスをそのまま日本語化して表示する
- `pipeline.editing` → 編集ステータスの大枠（`in_progress` / `not_scheduled` など）。**詳細な作業日程（何日の何時から）はここには無いため、手順3のカレンダー突き合わせで補う**
- `pipeline.publishDate` → 実績投稿日。値があればそれを「公開日（実績）」として採用し、`null`なら未公開として空欄にする

`data/stations.json` の値と実際の `stations/` 配下のファイル内容に食い違いがありそうな場合（例：`script.md`に加筆されているのに`pipeline.script`が`todo`のまま）は、進捗判定を`data/stations.json`の値で行いつつ、レポート内で「`data/stations.json`の更新が必要かもしれない」と一言添える（本スキルの対象外である`data/stations.json`自体の書き換えは行わない）。

### 3. カレンダーとの突き合わせ

Google Calendar MCP（`mcp__Google_Calendar__list_events`）をロードし、道の駅名を含む予定（`◯◯ 訪問`／`◯◯ 動画編集`／`◯◯ 動画公開`）を検索する。今日以降だけでなく、直近1〜2ヶ月の範囲で検索して既に完了した予定も含めて確認する。

- 見つかった `◯◯ 動画公開` の日時 → その道の駅の「予定投稿日」とする
- 見つからない場合は「予定投稿日：未登録」とする
- `◯◯ 動画編集` の予定が見つかれば、ステータス欄に編集作業の日程を反映する（例：「編集中（7/20-7/24 22:00-24:00 予定）」）

### 4. 週次投稿カレンダーの更新

2026-07-31週を起点とした週次の投稿枠を並べ、カレンダーに登録済みの `◯◯ 動画公開` 予定があればその週に道の駅を割り当てる。登録がない週は「未アサイン」とする。**割り当てを推測で埋めない**（カレンダーに予定がない週は必ず「未アサイン」と書く）。

### 5. 遅延判定

- 予定投稿日が今日より前で、かつ実績投稿日が空 → 「遅延」としてアラート扱いにする
- 直近の投稿予定週（次の日曜までなど）に予定投稿日が未登録 → 「要スケジュール登録」としてアラート扱いにする

### 6. `docs/progress.md` の更新

以下の構成で `docs/progress.md` を上書き生成する（最終更新日を実行日に更新）。

1. 見出しと管理方針の要約（本SKILL.mdの「管理方針」を簡潔に記載）
2. 「投稿ステータス一覧」テーブル：道の駅／訪問日／リサーチ／ルート／台本／編集予定／公開日(実績)／ステータス
3. 「週次投稿カレンダー（予実）」テーブル：投稿予定週／予定投稿日／担当の道の駅／実績投稿日／遅延
4. 「遅延・アラート」セクション：手順5で検出した項目を箇条書き。何もなければ「現時点で遅延・アラートなし」と明記する

既存の `docs/progress.md` がある場合は全体を再生成してよい（このファイルは`data/stations.json`とカレンダーから毎回再構成される派生データであり、手動加筆を前提としない）。

### 7. 完了報告

- 全体サマリ（進捗中の道の駅、直近の投稿予定、遅延の有無）を報告する
- 遅延・要対応事項があれば冒頭で強調する
- カレンダーに投稿予定が登録されていない週があれば、`schedule-station` での登録を提案する

## スコープ外

- Googleカレンダーの予定作成・変更（`schedule-station` の担当）
- `research.md`／`route.md`／`script.md` の中身の作成（`research-station` や通常の作業依頼で行う）
- コミット・プッシュ・PR作成は、ユーザーの指示があるまで行わない
